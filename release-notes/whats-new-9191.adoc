[[nas]]
== NAS

[cols="30%,70%"]
|===
| Aggiornare | Descrizione 


 a| 
xref:../flexcache/index.html[miglioramento delle performance NAS]
 a| 
ONTAP 9.19.1 consente di allocare maggiori risorse CPU per elaborare le richieste di elenco di directory NAS di piccole dimensioni. Questo miglioramento è vantaggioso per i clienti con centinaia o migliaia di client che elencano simultaneamente la stessa directory NAS di piccole dimensioni, pari o inferiore a 2 MB (circa 25.000 file).

Questo miglioramento riduce la necessità di ricorrere a soluzioni alternative, come volumi FlexCache locali, per supportare un elevato numero di client che eseguono elenchi di directory. Per un numero inferiore di client, le performance non saranno influenzate negativamente. Questo miglioramento delle performance è automatico e non richiede configurazione.

[[networking]]
== Networking

[cols="30%,70%"]
|===
| Aggiornare | Descrizione 


 a| 
Gestisci le versioni NFS supportate dalla SVM
 a| 
Il supporto per NFSv4.2 viene gestito indipendentemente dalle impostazioni di NFSv4.1, consentendo a ciascun SVM di pubblicizzare o disabilitare le funzionalità NFSv4.2 a seconda delle necessità. Per impostazione predefinita, NFSv4.2 è abilitato quando viene creato un nuovo servizio NFS, a meno che non venga esplicitamente disabilitato dall'amministratore.

* xref:../nfs-admin/enable-disable-nfs-task.html[Abilitare o disabilitare NFSv4.2 per SVM ONTAP]
* https://docs.netapp.com/us-en/ontap/task_nas_enable_linux_nfs.html["Abilitare l'archiviazione NAS per server Linux utilizzando NFS con ONTAP System Manager"]




 a| 
xref:../concepts/path-failover-concept.html#nas-path-failover[Miglioramento del failover LIF]
 a| 
La policy di gruppo di failover definita dal sistema ONTAP espande dinamicamente le destinazioni di failover per i LIF dati NAS, includendo potenzialmente tutte le porte disponibili nel dominio broadcast su qualsiasi nodo disponibile nel cluster. È possibile applicare la policy definita dal sistema a qualsiasi gruppo di failover. Prima di ONTAP 9.19.1, i LIF con questa policy potevano eseguire il failover solo su un altro nodo del cluster; se tale nodo non era in quorum, il LIF non eseguiva il failover, anche se erano disponibili altri nodi.



 a| 
link:../networking/improve-tcp-wan-performance.html["Miglioramenti delle prestazioni TCP"]
 a| 
ONTAP migliora le prestazioni del trasferimento dati TCP su collegamenti WAN con perdita di pacchetti e latenza elevata, integrando due funzionalità di recupero delle perdite, PRR e RACK, nello stack TCP di ONTAP. PRR è adatto a tutte le reti ed è abilitato per impostazione predefinita. RACK è adatto a reti congestionate con velocità di collegamento di circa 25 Gigabit/secondo ed è disabilitato per impostazione predefinita.

[[san]]
== SAN

[cols="30%,70%"]
|===
| Aggiornare | Descrizione 


 a| 
xref:../san-config/configure-fc-nvme-hosts-ha-pairs-reference.html[Supporto per configurazioni FC ad aggancio diretto]
 a| 
Gli switch FC non sono necessari per connettere host FC e FC-NVMe allo storage ONTAP. Gli host possono connettersi direttamente alle porte degli adattatori FC sui sistemi AFF, ASA e FAS. Questa connessione diretta è particolarmente utile nei siti remoti e negli ambienti ibridi dove la semplicità è importante.



 a| 
xref:../san-admin/multipath-automatic-lif-failover-support.html[Supporto per il multipathing attivo-attivo sui sistemi AFF]
 a| 
I sistemi AFF supportano il multipathing attivo-attivo negli ambienti SAN. Nelle configurazioni attivo-attivo, tutti i percorsi all'interno della coppia HA che possiede la LUN o il namespace sono ottimizzati in modalità attiva, in modo che in caso di failover dello storage, l'host non debba attendere la transizione ALUA o ANA dei percorsi di failover per riprendere l'I/O. Ciò riduce i tempi di failover, consentendo ai clienti SAN di eseguire carichi di lavoro mission-critical sui sistemi AFF con la stessa resilienza dei percorsi non interruttiva e bilanciamento del carico precedentemente disponibili solo sui sistemi ASA.



 a| 
xref:../san-admin/multipath-automatic-lif-failover-support.html[Supporto migliorato per il failover automatico di SAN LIF]
 a| 
Il failover automatico LIF è supportato sui sistemi AFF e ASA per i LIF del protocollo iSCSI in configurazioni multipath attivo-attivo e attivo-attivo-solo iSCSI, e per i LIF del protocollo NVMe/TCP in configurazioni multipath attivo-attivo.



 a| 
link:https://docs.netapp.com/us-en/ontap-fli/get-started/learn-about-data-migration.html["Supporto per l'importazione di LUN esterne per il protocollo iSCSI"^]
 a| 
Foreign LUN Import (FLI) supporta il protocollo iSCSI per la connettività backend.

[[security]]
== Sicurezza

[cols="30%,70%"]
|===
| Aggiornare | Descrizione 


 a| 
Le caratteristiche del carico di lavoro ARP sono state rimosse dall'interfaccia utente di System Manager
 a| 
System Manager non visualizza più le caratteristiche del carico di lavoro di Autonomous Ransomware Protection (ARP), incluse le statistiche di baseline e di picco. Queste informazioni sono state rimosse poiché l'attività di picco veniva spesso interpretata erroneamente come un vero e proprio attacco ransomware o come qualcosa che richiedeva un intervento manuale.

Il comportamento e il rilevamento ARP rimangono invariati, e solo la presentazione di questi dettagli in System Manager è interessata. Le statistiche di surge restano disponibili usando le opzioni ONTAP CLI e ONTAP REST API per un'analisi avanzata.



 a| 
xref:../networking/configure-tls-offload.html[Configurazione di offload TLS]
 a| 
È possibile configurare l'offload TLS per migliorare le prestazioni post-handshake TLS sfruttando le risorse sulle schede Ethernet supportate. Questa funzionalità esegue l'offload della crittografia e decrittografia, riducendo il sovraccarico della CPU e migliorando le performance.



 a| 
xref:../nfs-admin/enable-nfs-over-tls-task.html[Abilita NFS su TLS per crittografare il traffico NFS in transito]
 a| 
Puoi abilitare NFS over TLS con TLS 1.3 per crittografare il traffico NFS in transito sulle SVM abilitate per NFS. Questo miglioramento aiuta a proteggere i dati NFS da intercettazioni o manomissioni su reti non attendibili. La release supporta anche l'autenticazione host TLS reciproca opzionale per singola LIF e i controlli delle policy di esportazione per negare l'accesso TCP non TLS ai client corrispondenti.



 a| 
xref:../anti-ransomware/use-cases-restrictions-concept.html#snapmirror[Supporto per la protezione autonoma contro i ransomware per SnapMirror synchronous e SnapMirror active sync]
 a| 
ARP/AI protegge i volumi primari nelle relazioni SnapMirror sincrone (NAS e SAN) su FAS, AFF e  https://docs.netapp.com/us-en/ontap-afx/release-notes/whats-new-9191.html["AFX"] sistemi, e nelle relazioni SnapMirror active sync SAN su AFF e  https://docs.netapp.com/us-en/asa-r2/release-notes/whats-new-9191.html["ASA r2"] sistemi. Il rilevamento viene eseguito sul primario attivo in lettura-scrittura, dove vengono creati gli snapshot ARP (gli snapshot non vengono replicati).

I cluster aggiornati a ONTAP 9.19.1 che in precedenza avevano la funzione ARP predefinita disabilitata a causa di relazioni di replica sincrona la vedranno ripristinata automaticamente. Un avviso pre-aggiornamento avverte gli amministratori e offre la possibilità di disabilitarla.



 a| 
xref:../authentication-access-control/webauthn-mfa-overview.html[WebAuthn miglioramento del dominio della Relying Party]
 a| 
ONTAP 9.19.1 introduce un nuovo parametro  `-rp-domains` nel comando delle impostazioni globali WebAuthn, che ti permette di limitare esplicitamente l'insieme dei domini Relying Party (RP) consentiti durante l'autenticazione a più fattori WebAuthn (FIDO2) in System Manager.
